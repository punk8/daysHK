# AGENTS.md

## 项目背景

本项目是一个用于记录用户在香港停留天数的移动应用，暂定名称为「在港日记」或「Days in HK」。

目标用户是准备申请香港永久居民身份的大陆居民。他们需要长期记录自己进入香港、离开香港的日期，并按年份统计估算在港天数，辅助个人时间管理和申请材料准备。

产品定位是「个人在港记录与估算统计工具」，不是法律资格判断工具。所有涉及永居、通常居住、连续离港等内容，都应使用谨慎表达，避免承诺用户符合或不符合香港永久居民资格。

## 核心功能

MVP 需要优先覆盖：

- 首页 Dashboard：展示当前状态、今年估算在港天数、连续在港 / 离港信息、最近记录和权限提示。
- 年度统计：按香港自然日统计年度在港天数，展示月度分布和连续离港提醒。
- 入离港记录：展示入港、离港记录，区分自动检测、用户确认、手动补录和需要确认状态。
- 手动补录：允许用户手动新增入港 / 离港记录，支持当天往返、地点、交通方式和备注。
- 设置与隐私：展示定位权限、本地存储说明、CSV 导出、隐私说明、清除本地数据等。

后续增强功能可以包括：

- iOS / Android 自动定位和地理围栏打卡。
- 香港边界 polygon 判断。
- PDF 报告导出。
- iCloud / Google Drive / 云同步。
- 多语言支持。

## 已确认产品规则

- 第一版同时支持 iOS 和 Android。
- 第一版采用本地存储，不做账号系统，不做云同步。
- 第一版主界面按「天 / 日期」统计，不以小时作为核心展示单位。
- 统计统一按香港本地自然日计算。
- 入港当天计入在港日。
- 离港当天计入在港日。
- 当天往返香港计为 1 个在港日。
- 同一天多次入离港，去重后仍只计为 1 个在港日。
- 跨月、跨年度停留按自然日拆分到对应月份和年份。
- 离港期间按完整不在港自然日计算，离港日和返港日不计为完整离港日。
- 连续离港超过 6 个月时，应提示用户该期间可能需要解释或留证。

## 技术方案

当前建议技术栈：

- App 框架：Flutter
- 本地数据库：SQLite，优先考虑 drift 或 sqflite
- iOS 定位能力：Core Location
- Android 定位能力：Kotlin + Geofencing API / Fused Location Provider
- 原生扩展方式：Flutter Platform Channels
- 边界判断：内置香港边界 MultiPolygon，运行时做 bbox 粗过滤 + point-in-polygon 判断
- 数据导出：CSV 作为 MVP 优先项

自动定位不应成为 MVP 的唯一价值基础。开发顺序建议先实现手动记录、SQLite 存储、日期统计和记录列表，再接入自动定位、地理围栏和边界判断。

后续实现时应遵循该 UI 的整体方向：

- 专业、可信、安静，不做旅游 App 或营销页风格。
- 以浅色背景、白色卡片、深色文字为主，搭配克制的香港红和青绿色 / 蓝绿色强调色。
- 页面结构以首页、统计、记录、补录、设置五个主视图为基础。
- 首页突出当前状态和今年估算在港天数。
- 记录页要能表达自动检测、用户确认、手动补录、需要确认等来源和状态。
- 手动补录页要显示本次将计入多少个在港日。
- 设置页要强调本地数据存储、导出和隐私说明。

## UI 需要补齐的状态

当前 UI 示意图可以作为开发起点，但开发时需要补齐以下状态：

- 首页的「不在香港」状态。
- 首页的「需要确认」状态。
- 更细的定位权限异常状态，例如未开启始终允许、后台刷新关闭、省电模式可能影响记录。
- 记录页中「需要确认」记录的确认 / 修正操作。
- 手动补录页中离港日期为空时的含义：代表当前仍在香港。
- 离港日期早于入港日期时的校验状态。
- 清除所有数据的二次确认。
- 连续离港提醒文案应使用「可能需要解释或留证」，避免强法律判断。

## 数据与隐私要求

- 默认所有数据保存在本地设备。
- MVP 不上传、不同步、不共享用户数据。
- 用户应能导出 CSV。
- 用户应能清除本地数据，但必须二次确认。
- 涉及定位、出入境、永居申请的文案应谨慎、透明、可解释。

## 开发注意事项

- 优先实现可用 MVP，不要一开始过度抽象。
- 不要把 App 设计成永居资格判定器。
- 自动定位生成的记录应尽量作为候选记录或可解释记录，允许用户确认和修正。
- 边界附近的定位结果应支持「需要确认」状态，不应绝对判断。
- 所有统计结果都应保留可追溯记录，用户能看到每个天数来自哪些入离港记录。
